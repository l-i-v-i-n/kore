# kore

Convert JSON values into **`.kore`** — a compact, human-readable text format
for representing API responses and structured data — without losing any
information from the original JSON.

This is a Rust port of [`kore-js`](https://github.com/kore-lang/kore-js),
built to be output-compatible for the common cases (flat objects, tables,
lists, scalars, envelope-style API responses), while fixing a few real bugs
found in the reference implementation along the way. See
[Differences from kore-js](#differences-from-kore-js) below for the full,
honest list.

```toml
[dependencies]
kore = "1.0"
```

## Quick start

```rust
use kore::{to_kore, KoreOptions};
use serde_json::json;

let hikes = json!([
    { "id": 1, "name": "Blue Lake Trail", "km": 7.5, "sunny": true },
    { "id": 2, "name": "Ridge Overlook",  "km": 9.2, "sunny": false },
]);

let result = to_kore(&hikes, &KoreOptions::new("hikes"));
println!("{}", result.kore);
```

```text
hikes[id, name, km, sunny]:
  1 | Blue Lake Trail | 7.5 | true
  2 | Ridge Overlook  | 9.2 | false
```

`result` also carries a bit of metadata about what was detected, so callers
don't need to re-parse the text (continuing the example above):

```rust
# use kore::{to_kore, KoreOptions};
# use serde_json::json;
# let hikes = json!([
#     { "id": 1, "name": "Blue Lake Trail", "km": 7.5, "sunny": true },
#     { "id": 2, "name": "Ridge Overlook",  "km": 9.2, "sunny": false },
# ]);
# let result = to_kore(&hikes, &KoreOptions::new("hikes"));
assert_eq!(result.structure.as_str(), "table");
assert_eq!(result.row_count, Some(2));
assert_eq!(result.columns, Some(vec![
    "id".to_string(), "name".to_string(), "km".to_string(), "sunny".to_string(),
]));
```

## Converting a JSON string directly

```rust
use kore::{to_kore_from_str, KoreOptions};

let result = to_kore_from_str(
    r#"[{"id":1,"name":"ana"}]"#,
    &KoreOptions::new("users"),
).expect("valid json");
```

`to_kore_from_str` returns `Result<KoreResult, serde_json::Error>`, so a
malformed JSON string is a normal `Err`, not a panic.

## Type inference

Turning on `infer_types` adds an `@types { ... }` block above a table,
inferring a narrow type per column straight from the data:

```rust
use kore::KoreOptions;

let opts = KoreOptions::new("hikes").infer_types(true);
```

```text
@types {
  Hikes {
    id: u8
    name: str
    km: f32
    sunny: bool
  }
}

hikes[id, name, km, sunny]:
  1 | Blue Lake Trail | 7.5 | true
  2 | Ridge Overlook  | 9.2 | false
```

Numeric columns are widened to the smallest type that fits every value seen
(`u8` → `u16` → `u32` → `u64`, or `i32` → `i64` for negative numbers, or
`f32`/`f64` for non-integers); a column is marked nullable (`type?`) if any
row was missing the field or had it explicitly set to `null`; strings that
look like `YYYY-MM-DD` or `http(s)://...` are tagged `date` and `url`.

## Metadata and comments

```rust
use kore::KoreOptions;

let opts = KoreOptions::new("hikes")
    .comment("generated by kore")
    .with_ctx("source", "trail-api")
    .with_ctx("page", 1);
```

```text
// generated by kore
ctx(source=trail-api, page=1)

hikes[...]:
  ...
```

## Envelope-style API responses

A very common shape for API responses is an envelope with one metadata
field and one data field, e.g. `{ "status": "ok", "page": 1, "hikes": [...] }`.
When `to_kore` sees a plain object with more than one key where *exactly
one* key holds an array or nested object, it automatically treats that key
as "the data" and renders the rest as a `ctx(...)` line above it:

```rust
use kore::{to_kore, KoreOptions};
use serde_json::json;

let api_response = json!({
    "status": "ok",
    "page": 1,
    "hikes": [
        { "id": 1, "name": "Blue Lake", "km": 7.5 },
        { "id": 2, "name": "Ridge",     "km": 9.2 },
    ]
});

let result = to_kore(&api_response, &KoreOptions::new("hikes"));
```

```text
ctx(status=ok, page=1)

hikes[id, name, km]:
  1 | Blue Lake | 7.5
  2 | Ridge     | 9.2
```

If an object has **more than one** nested (array or object) field, there's
no single "data key" to extract, so it falls back to rendering every field
in full — see the next section for why that matters.

## Differences from kore-js

This port intentionally matches kore-js's output for ordinary cases, but
fixes three issues found while reading through the reference implementation
closely enough to reimplement it. None of these change output for the
common shapes shown above; they only change behavior in cases that were
previously incorrect or ambiguous.

### 1. Fixed: silent data loss on objects with multiple nested fields

This was the most serious one. In kore-js, the "envelope" detection picks
the *first* key whose value is an array or object as "the data," but it
does this unconditionally whenever an object has more than one key —
including when the object has **two or more** nested fields. Every nested
field after the first one was silently dropped, with no trace in the
output (it doesn't even show up as `ctx` metadata, since `ctx` only
collects scalar values).

```rust
use kore::{to_kore, KoreOptions};
use serde_json::json;

let data = json!({
    "name": "ana",
    "address": { "city": "Boulder" },
    "tags": ["fast", "fun"]
});
```

A faithful one-to-one port of kore-js would render only `address` and
silently lose `tags` entirely. This crate checks how many nested fields
exist first, and only takes the envelope shortcut when there's exactly
one; otherwise it falls back to rendering every field:

```text
name: ana
tags: [fast, fun]

address(city=Boulder)
```

### 2. Fixed: `@types` block could disagree with the table header

kore-js's type inference only looked at the *first* row's keys to decide
which columns exist, while the table renderer itself unions keys across
*all* rows. If row 0 happened to be missing a field that a later row had,
that column would silently vanish from the `@types` block while still
appearing in the table — two blocks in the same document disagreeing about
the schema. This port uses the same column union for both, so they always
agree.

### 3. Improvement: array values are bracketed wherever they're inlined

kore-js renders an array field inline as a bare comma-joined list with no
delimiter, e.g. a `tags` field next to other fields could render as
`tags=a, b, c` — visually indistinguishable from three separate
comma-separated fields. This port wraps inlined arrays in `[...]`
(`tags=[a, b, c]`), which is unambiguous and still trivial to parse back
out.

Everything else — table layout, padding, `ctx()` syntax, null handling
(`_`), pipe-delimited rows, quoting rules for ambiguous table cells, the
`@types` block syntax — matches kore-js exactly, so `.kore` output from
this crate and from kore-js can be read by the same downstream tooling.

## API overview

| Item | Description |
|---|---|
| `to_kore(&Value, &KoreOptions) -> KoreResult` | Convert an already-parsed `serde_json::Value`. |
| `to_kore_from_str(&str, &KoreOptions) -> Result<KoreResult, serde_json::Error>` | Parse a JSON string, then convert. |
| `KoreOptions` | Builder-style options: `block_name`, `ctx`, `infer_types`, `indent`, `comment`, `max_col_width`. |
| `KoreResult` | `{ kore, structure, columns, row_count }` — the rendered text plus what was detected. |
| `Structure` | `Table \| Object \| List \| Scalar`. |
| `serialize_value`, `ctx_value` | Lower-level value-rendering functions, exported in case you're building your own renderer on top. |
| `infer_value_type`, `infer_table_types`, `build_types_block` | Type-inference building blocks, also exported standalone. |

Full API docs (with every function and example) are generated from doc
comments — run `cargo doc --open` after adding the dependency, or browse
docs.rs once published.

## Development

```bash
cargo build
cargo test          # unit tests + integration tests + doctests
cargo clippy --all-targets
```

The integration test suite in `tests/converter_tests.rs` mirrors the
original kore-js Jest suite case-for-case (plus the regression tests for
the fixes above), so it's easy to confirm behavior at a glance.

## Publishing

The crate is named `kore` locally for development, but **`kore` is already
taken on crates.io** by an unrelated project. Before running `cargo
publish` for the first time, change `name` in `[package]` (and to match,
`name` in `[lib]`) to something free — `kore-format` was free as of this
writing, but double-check at crates.io/crates/&lt;name&gt; before settling
on one. Every `use kore::...` in your own downstream code would then become
`use <new-name>::...`.

```bash
cargo login <your-crates-io-token>
cargo publish --dry-run   # sanity check the package contents
cargo publish
```
